# .principles audit context — pipeline
# Sections: Principle · Violations to detect

### PIPELINE-MINIMAL-PERMISSIONS — Run pipeline jobs with minimal permissions

Every pipeline job should run with the minimum permissions required to complete its specific task. Broad IAM roles, wildcard scopes, or reused service accounts with accumulated permissions violate this principle.

Violations to detect:

- A single pipeline service account with admin or broad write access used across all jobs
- `permissions: write-all` or absent `permissions:` block (defaults to write) in GitHub Actions workflows
- IAM roles with `Action: "*"` or `Resource: "*"` assigned to pipeline execution roles
- Secrets with broad scope passed to all pipeline steps
- Deploy steps that receive the same credentials as build steps

### PIPELINE-NO-SECRETS-IN-LOGS — Never expose secrets in pipeline logs

Secrets, tokens, passwords, and credentials must never appear in CI/CD pipeline logs or job output.

Violations to detect:

- `echo $DATABASE_PASSWORD` or `print(os.environ["SECRET"])` in pipeline scripts
- Verbose curl/wget commands with `-v` flag that print request headers including `Authorization:` tokens
- Commands that expand secrets inline: `mysql -u root -p$DB_PASS`
- `set -x` bash debug mode active globally in scripts that handle secrets
- Environment variable dumps (`env`, `printenv`) that run when secrets are in scope

### PIPELINE-REPRODUCIBLE-BUILDS — Ensure builds produce identical outputs from identical inputs

A pipeline build is reproducible when the same source code, dependency set, and build toolchain produce a bit-for-bit identical artifact every time. Achieve this by pinning dependency versions in lockfiles, pinning build tool versions, avoiding network fetches during compilation, and eliminating reliance on ambient machine state.

Violations to detect:

- `npm install` instead of `npm ci` or `yarn install --frozen-lockfile` in build steps
- `pip install -r requirements.txt` without `--require-hashes` or a lockfile
- Missing lockfiles in the repository
- Build steps that download tools at runtime (`curl | bash`) without version pinning or checksum verification
- Use of `latest` tag for build tool images
- Timestamps, random UUIDs, or build-machine hostnames embedded in artifact metadata

### PIPELINE-ENVIRONMENT-ISOLATION — Pipeline stages must not share mutable state

Each pipeline stage must execute in an isolated context with its own workspace, credentials, and environment variables. Stages must not share mutable filesystem state, database connections, or environment variables beyond what is explicitly passed as stage outputs.

Violations to detect:

- Shared filesystem mounts or working directories between build and deploy stages without explicit artifact passing
- Environment variables with production credentials available to all stages including build and test
- Pipeline steps that write to a shared global cache used by subsequent steps without scoping
- Reuse of the same container instance across stages rather than starting fresh per stage
- `services:` or `docker-compose` blocks that persist across unrelated stages

### PIPELINE-FAIL-FAST-PIPELINE — Abort the pipeline on the first meaningful failure

When a pipeline stage fails, subsequent dependent stages must not execute. Allow-failure and continue-on-error directives must be used sparingly and only for explicitly non-blocking checks.

Violations to detect:

- `continue-on-error: true` on build, test, or deploy steps in GitHub Actions
- `allow_failure: true` on critical stages in GitLab CI
- `|| true` or `set +e` appended to commands in pipeline scripts to suppress failures
- Deploy stages that run unconditionally regardless of test stage outcomes
- Missing stage dependencies — stages run in parallel when they should be sequential with failure propagation
- `try/catch` blocks in Jenkinsfile that silently swallow stage failures

### PIPELINE-DEPLOYMENT-GATES — Require explicit approval or conditions before production deployment

Deployment to production or other protected environments must pass through an explicit gate — a manual approval, an automated policy check, or both — enforced by the pipeline platform, not by convention.

Violations to detect:

- Production deployment steps with no `environment:` declaration or protection rules
- Deploy-to-production jobs triggered directly by push events with no approval gate
- Missing `required_reviewers` on GitHub Environment protection rules for production
- GitLab deploy stages to production without `when: manual` or `protected: true` environment
- Deployment triggered from any branch rather than restricted to `main`/`release` branches
- No distinction between staging and production deployment conditions
