# .principles audit context — sec-arch
# Sections: Principle · Violations to detect

### SEC-ARCH-THREAT-MODELLING — Model threats before implementing security controls

Before implementing any security-sensitive feature, systematically enumerate threats using a structured methodology such as STRIDE or PASTA. Identify trust boundaries, data flows, threat actors, and attack vectors; document mitigations; and produce a durable artifact reviewed alongside the design.

Violations to detect:

- No threat model documented for features that handle authentication, authorisation, payments, or PII
- Trust boundaries not identified — components operating at different trust levels communicate without explicit validation
- Security design review conducted after implementation decisions are locked in, when changes are costly
- Threats discussed verbally during design but not recorded, tracked, or verified during implementation
- Data flow diagrams absent — it is not possible to reason about what crosses a trust boundary

### SEC-ARCH-ZERO-TRUST — Never trust implicitly; verify every request

Never grant implicit trust based on network location. Every request must be authenticated with a verified identity, authorised for the specific resource, and transmitted over an encrypted channel. Trust is established per-request through cryptographic verification, not inherited from network location.

Violations to detect:

- Service-to-service calls within the same VPC, subnet, or Kubernetes cluster that skip authentication on the assumption that "it's internal traffic"
- Authorisation checks absent on internal API endpoints that are not publicly routable
- Unencrypted inter-service communication (plain HTTP) on the assumption that the internal network is private
- Identity established by source IP address or network segment rather than cryptographically verified credentials
- Long-lived service credentials (API keys, passwords) without automatic rotation
- No network segmentation between services — any compromised workload can reach any other workload on all ports

### SEC-ARCH-SUPPLY-CHAIN-SECURITY — Verify provenance and integrity across the build chain

Treat every external dependency, container base image, build tool, and CI action as a potential attack vector. Verify integrity through cryptographic signatures and provenance attestations; generate and publish SBOMs; and structure build pipelines to be hermetically sealed and reproducible.

Violations to detect:

- No SBOM generated or published for released artifacts
- Container base images referenced by mutable tag rather than by immutable digest (`sha256:...`)
- Build pipelines that fetch dependencies from the public internet at build time without a private artifact proxy
- Third-party CI actions referenced by a floating branch or tag rather than a pinned commit SHA
- No cryptographic signature verification of downloaded artifacts before use in the build
- Build pipeline steps with broad outbound network access during compilation

### SEC-ARCH-ECONOMY-OF-MECHANISM — Keep security mechanisms as simple as possible

Security mechanisms must be as simple and small as possible so that they can be correctly implemented, thoroughly tested, and confidently reasoned about. Every line of security-critical code is a potential source of vulnerability; complexity in auth logic multiplies the states an attacker can exploit.

Violations to detect:

- Custom cryptographic algorithms or hand-rolled protocol implementations instead of established libraries
- Authentication or authorisation logic scattered across multiple modules with overlapping, inconsistent rules
- Access control lists with redundant, conflicting, or deeply nested rules
- Security middleware with multiple code paths for the same decision, some dead or untested
- Token validation logic that manually parses JWTs rather than using a vetted library
- Complex session management with custom logic when a standard framework is available

### SEC-ARCH-SEPARATION-OF-PRIVILEGE — Require multiple independent conditions to grant access

Access to a protected resource or operation must depend on satisfying two or more independent conditions. No single credential, approval, or factor should be sufficient to perform a sensitive action.

Violations to detect:

- Authentication that accepts a single factor for sensitive operations without requiring a second independent factor
- Deployment pipelines where a single developer can merge and deploy to production without a second approval
- Administrative actions that require only a single authenticated session — no re-authentication, MFA, or countersignature
- API keys or service accounts that grant broad access without a secondary authorisation token
- Secret management systems where a single identity can read production secrets without a co-signer
- Infrastructure-as-code changes that can be applied by a single actor without peer review

### SEC-ARCH-LEAST-COMMON-MECHANISM — Minimize shared mechanisms between users and components

Minimize the amount of mechanism shared between users, tenants, or components that operate at different trust levels. Shared service accounts, global singletons for auth state, and common resources create covert channels and blast-radius amplifiers.

Violations to detect:

- A single service account shared across multiple applications or environments at different trust levels
- A shared in-memory authentication or session cache accessible to multiple tenants without isolation
- Multiple services writing to the same database schema without tenant isolation
- A shared file system or storage bucket used by services at different trust levels without access-control boundaries
- A single logging pipeline commingling data from different tenants or security zones
- Global singletons or static variables holding security state in multi-tenant applications

### SEC-ARCH-PSYCHOLOGICAL-ACCEPTABILITY — Security must not make the system harder to use correctly

Security mechanisms must be designed so that the secure path is the easiest path. If a security control makes the system harder to use correctly than to use insecurely, users and developers will find workarounds that defeat the control.

Violations to detect:

- Security configuration requiring editing multiple files across different locations for a single logical operation
- Secret management APIs significantly more complex than hardcoding the value
- Permission prompts appearing so frequently that users develop click-through habits
- Authentication flows requiring more than 2-3 steps for routine access
- Security-critical APIs where the insecure usage pattern requires fewer lines of code than the secure pattern
- Security documentation that exists only as a dense monolithic reference

### SEC-ARCH-OPEN-DESIGN — Security must not depend on secrecy of the mechanism

The security of a system must not depend on the secrecy of its design, algorithms, or implementation. Assume that an attacker has full knowledge of the system's architecture and source code. Security must derive from the secrecy of keys and credentials, not from obscurity of the mechanism.

Violations to detect:

- Hardcoded secret values used as the primary security barrier
- Custom or proprietary encryption algorithms instead of peer-reviewed standards
- Security depending on URL obscurity — unprotected endpoints that are "hidden" but not authenticated
- Obfuscation of code or configuration treated as a security control
- Internal API endpoints with no authentication because they are "not publicly documented"
- Comments or documentation stating that security depends on attackers not knowing how the mechanism works
