# .principles audit context — docs
# Sections: Principle · Violations to detect

### DOC-TASK-ORIENTED — Organise documentation around user tasks, not product features

Documentation should be structured around what users need to accomplish, not around how the product is organised internally. Headings and sections should reflect user goals expressed as tasks, not product components.

Violations to detect:

- Top-level navigation or section headings that mirror the product's module or component names rather than user goals
- Headings phrased as nouns (`"Configuration"`, `"Authentication"`, `"Permissions"`) where the section is procedural and could be a task verb
- A table of contents that lists features exhaustively but provides no grouping by user scenario or task
- Reference material mixed into task-oriented sections, forcing users to parse both types to extract the procedure
- Getting-started guides structured as a feature tour rather than a sequence of tasks producing a working result

### DOC-SCANNABLE — Structure documents so readers can navigate without reading every word

Readers of technical documentation scan for the section relevant to their current goal. Documentation should support scanning through descriptive headings, short paragraphs, bullet lists, and bold text anchoring key terms.

Violations to detect:

- Paragraphs longer than 5–6 sentences without a visual break
- Sections of more than 300 words with no heading, subheading, or list
- A list of three or more parallel items written as a comma-separated sentence rather than a bulleted list
- Headings that are questions or vague labels (`"More details"`, `"Overview"`) that don't tell the scanner what the section contains
- Step-by-step instructions written as prose rather than a numbered list

### DOC-OBJECTIVE — Write in a neutral, factual tone; eliminate promotional language

Technical documentation should be factual and neutral. Avoid adjectives that express enthusiasm or ease without evidence: `"easy"`, `"simple"`, `"powerful"`, `"seamlessly"`, `"robust"`.

Violations to detect:

- Unsupported quality adjectives: `"easy"`, `"simple"`, `"intuitive"`, `"powerful"`, `"robust"`, `"flexible"`, `"best"`, `"seamlessly"`, `"effortlessly"`
- Marketing framing in technical content: `"our industry-leading API"`, `"world-class performance"`
- Unverifiable comparative claims with no benchmark or citation
- Instructional language implying the reader is wrong for not knowing: `"simply run"`, `"just add"`, `"obviously"`

### DOC-SELF-CONTAINED — Every page must make sense without requiring other pages to be read first

Every documentation page should be intelligible to a reader who arrives directly via search or a shared link, without having read any other page.

Violations to detect:

- A page that uses an acronym or term on first use without defining it or linking to a definition
- A how-to guide whose first step assumes the reader completed steps on a different page, with no link
- A section beginning `"As described above…"` where the reference is to a different document
- A page with no introductory sentence establishing what it is about and who it is for
- Tutorial steps that reference a configuration set up in a previous tutorial without re-stating the prerequisite

### DOC-ADDRESSABLE — Make every section directly linkable

Every meaningful section should be reachable by a stable, direct link. Use consistent headings so renderers generate anchor IDs, keep heading text stable once published, and avoid duplicate headings.

Violations to detect:

- Duplicate heading text within the same file (produces colliding anchor IDs)
- Headings too vague to produce useful anchors: `"Details"`, `"More"`, `"Notes"`, `"Other"`
- A long document with no headings, making no section linkable
- AsciiDoc or Pandoc Markdown sections without explicit anchor IDs where stable links are needed
- Internal cross-references using `"see below"` or `"see above"` rather than named section links

### DOC-UNIQUE — Each piece of information has exactly one home

Every piece of documentation content should exist in exactly one canonical location. Other documents that need that information should link to the canonical source, not copy its text.

Violations to detect:

- Installation instructions repeated verbatim in a `README.md`, a `CONTRIBUTING.md`, and a `docs/getting-started.md` with no cross-links
- A configuration reference whose parameter descriptions appear both in the API docs and in a separate guide, with no indication which is authoritative
- A changelog or version history maintained in both a `CHANGELOG.md` and a wiki page
- Conceptual background sections copy-pasted across multiple how-to guides rather than extracted into a shared explanation document
- Two documents that describe the same workflow in different words, creating ambiguity about which is current

### DOC-AS-CODE — Treat documentation as source code

Documentation should live in version control alongside the code it describes, be authored in a plain-text format, and follow the same contribution workflow as code: branches, pull requests, peer review, and automated validation in CI.

Violations to detect:

- A repository where documentation files are absent and a wiki URL is the only documentation reference in the README
- Documentation committed directly to the main branch with no pull request or review trail
- No automated validation (link checking, doc-tests, linting) configured for documentation in CI
- Binary or proprietary-format documentation files (`.docx`, `.pdf` source) committed to the repository
- A `CHANGELOG` or API documentation that is generated from a separate, unversioned source

### DOC-CLOSE-TO-CODE — Keep documentation close to what it describes

Documentation should live as close as possible to the artifact it describes. Distance between a document and its subject is the primary driver of documentation drift.

Violations to detect:

- A monorepo where all documentation is centralized in a top-level `docs/` directory with no documentation co-located in individual module or service directories
- A library or package with no `README.md` at its own directory level
- API documentation stored in a separate repository from the API implementation
- A directory containing complex logic with no adjacent explanation — no `README.md`, no doc-comment, no `ARCHITECTURE.md`
- Documentation that refers to file paths or module names that no longer exist in the adjacent source tree

### DOC-PURPOSE — Every document has one clear purpose

A document should serve exactly one reader goal. The four fundamental documentation types — tutorials, how-to guides, reference material, and explanations — serve different purposes and should not be mixed. If a document tries to be a tutorial and a reference simultaneously, it fails at both.

Violations to detect:

- A README that contains step-by-step installation instructions, an API reference, a conceptual overview, and a changelog all in one file
- A tutorial that interrupts the step sequence with exhaustive option tables
- An architecture doc that doubles as an operations runbook
- A how-to guide that starts with long explanatory background before the first actionable step
- No stated purpose or goal in the opening section

### DOC-MINIMAL — Write only what is necessary

Apply YAGNI to prose: do not document speculative features, rarely-used edge cases, or information the reader can derive from context. Every sentence has a maintenance cost.

Violations to detect:

- README files that document every configuration option when a link to reference docs would suffice
- Getting-started guides that begin with five paragraphs of history before the first actionable step
- Documentation that re-states what is obvious from the code, config, or type signature
- Hedge sentences like "this may or may not apply in your case" that add words without adding information
- Copy-pasted boilerplate sections that contain no content specific to this document

### DOC-AUDIENCE — Write for a specific, named audience

Every document should be written for a specific, identifiable reader with a known level of existing knowledge and a concrete goal.

Violations to detect:

- Documentation that alternates between beginner explanations and expert shortcuts in the same section
- Unexplained jargon in a getting-started guide targeted at newcomers
- Verbose background in reference material targeted at experienced users
- No stated audience or prerequisite knowledge at the top of a guide
- A single document covering both quick-start and advanced configuration without clear audience segmentation

### DOC-ACCURACY — Keep documentation accurate and synchronized with the system

Documentation must accurately describe the current state of the system it documents. Out-of-date documentation is worse than no documentation.

Violations to detect:

- Code examples in documentation that no longer compile or execute correctly
- API parameter names, types, or defaults that differ from the actual implementation
- Architecture diagrams that describe a previous design rather than the current one
- Instructions that reference a command, flag, or option that no longer exists or has changed
- Version numbers in documentation that lag behind the released version

### DOC-EXAMPLES — Illustrate with concrete, runnable examples

Abstract descriptions should be accompanied by concrete examples a reader can try immediately.

Violations to detect:

- API reference documentation that lists parameters with no example request/response pair
- A tutorial that describes configuration options in prose without showing a minimal working configuration file
- Code examples that use placeholder names like `foo`, `bar`, `data`, or `thing` that obscure the actual usage pattern
- Examples that demonstrate only error handling or edge cases, never the happy path
- Documentation that says "see the source code for examples" as a substitute for written examples

### DOC-PROGRESSIVE-DISCLOSURE — Reveal complexity progressively

Present information in layers of increasing complexity: begin with the simplest case that covers the majority of readers, then progressively reveal advanced options, edge cases, and deeper explanations.

Violations to detect:

- A getting-started guide that begins with a complete configuration reference before showing a minimal working example
- A tutorial that explains all edge cases and failure modes before the reader has seen the happy path succeed
- An architecture document that starts with implementation details before establishing the high-level mental model
- Advanced options embedded in the same section as basic options without clear visual separation
- No quick start or TL;DR section for readers who need the 20% that covers 80% of cases
