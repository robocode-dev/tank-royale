# .principles prime context — docs
# Sections: Principle · Why it matters · Good practice

### DOC-TASK-ORIENTED — Organise documentation around user tasks, not product features

Documentation should be structured around what users need to accomplish, not how the product is organised internally. Headings and sections should reflect user goals as task verbs, not product component nouns.

Why it matters:

Feature-oriented documentation forces users to understand the product's internal structure before finding what they need. Task-oriented documentation meets users at the point of a goal — reducing cognitive load and search time.

Good practice:

- Write headings as verb phrases: `"Configure rate limiting"` rather than `"Rate Limiting"`
- Group documentation by user scenario; keep feature reference in a clearly labelled separate section
- Validate structure by asking: can a new user read the table of contents and understand what they can accomplish?

### DOC-SCANNABLE — Structure documents so readers can navigate without reading every word

Readers scan technical documentation for the section relevant to their current goal. Support scanning with descriptive headings, short paragraphs, bullet lists, and bold key terms.

Why it matters:

Usability research (Morkes & Nielsen, 1997) measured a 47% improvement in usability from scannable structure. Poor scannability forces linear reading to find a single fact — directly increasing support burden.

Good practice:

- Use descriptive headings that name the concept or action, not generic labels
- Break any list of three or more parallel items into a bulleted list
- Keep paragraphs to one idea, 3–5 sentences; write the most important sentence first
- Use numbered lists for sequential steps; bullet lists for unordered sets

### DOC-OBJECTIVE — Write in a neutral, factual tone; eliminate promotional language

Technical documentation should be factual and neutral. Avoid adjectives expressing enthusiasm or ease without evidence: `"easy"`, `"powerful"`, `"seamlessly"`, `"simply"`.

Why it matters:

Morkes & Nielsen (1997) measured a 27% usability improvement from objective writing. Promotional language signals persuasion over information, causing readers to discount surrounding content and slow their reading.

Good practice:

- Replace `"simple to configure"` with a minimal configuration example — let the reader judge
- Remove `"simply run X"` → `"run X"`; the qualifier patronises readers who find it non-trivial
- Back comparative claims with a benchmark or citation; otherwise cut them
- Review PRs for promotional language as a distinct pass

### DOC-SELF-CONTAINED — Every page must make sense without requiring other pages to be read first

Every documentation page should be intelligible to a reader arriving directly — via search, a shared link, or an in-app reference — without having read any surrounding pages.

Why it matters:

On the web, every page is potentially the first page a reader sees. Documentation written for sequential reading fails every reader who arrives via a direct link, which in practice is most readers.

Good practice:

- Open every page with a sentence stating its purpose and intended reader
- Define or link every term a cold-arriving reader would not know
- State prerequisites explicitly at the top with links: `"Before you begin: complete X"`
- Replace `"as described above"` with a direct named section link when the reference is to another page

### DOC-ADDRESSABLE — Make every section directly linkable

Every meaningful section should be reachable by a stable, direct link. Use consistent, specific headings; avoid duplicates; add explicit anchors where stability is required.

Why it matters:

Technical documentation is constantly referenced in issues, PRs, chat, and runbooks. Sections without stable anchors break silently when headings are renamed, forcing readers to scan entire documents to find the referenced content.

Good practice:

- Use specific, descriptive heading text so auto-generated anchors are human-readable
- Avoid duplicate headings within a file; disambiguate where needed
- Add explicit anchor IDs (`{#id}`, `[[id]]`) to frequently-linked sections
- Replace `"see below"` / `"see above"` with named section links

### DOC-UNIQUE — Each piece of information has exactly one home

Every piece of documentation content should exist in exactly one canonical location. Other documents that need that information should link to the canonical source, not copy its text. When the same content lives in multiple places, each copy becomes an independent maintenance obligation that will silently diverge.

Why it matters:

Duplicated documentation creates silent inconsistency. When a procedure changes, the author updates the page they are thinking about and forgets the copies. Readers following an outdated copy encounter errors and lose trust in the documentation — with no compile error to flag the problem.

Good practice:

- Identify the single correct home for each piece of information; every other document links there instead of copying it
- Treat a duplicated section as a refactoring target: extract to canonical location, replace copies with links
- Use include directives where the toolchain supports them, to transclude rather than copy
- During PR review, search for content already documented elsewhere before approving additions

### DOC-AS-CODE — Treat documentation as source code

Documentation should live in version control alongside the code it describes, be authored in a plain-text format, and follow the same contribution workflow as code: branches, pull requests, peer review, and automated validation in CI.

Why it matters:

Documentation maintained outside version control drifts from the code it describes because updates to docs and code are not coupled. There is no audit trail, no diff, and no review gate. Over time, unreviewed content accumulates and trust in the documentation collapses.

Good practice:

- Store all documentation in version control in the same repository as the code it describes
- Use plain-text formats (Markdown, AsciiDoc, reStructuredText) that are diffable and mergeable
- Require documentation updates in the pull request that changes user-facing behavior
- Run link checkers, spell checkers, and doc-tests in CI on every PR that touches documentation

### DOC-CLOSE-TO-CODE — Keep documentation close to what it describes

Documentation should live as close as possible to the artifact it describes. A module's README belongs in the module's directory. Distance between a document and its subject is the primary driver of documentation drift.

Why it matters:

Documentation stored far from the code it describes is updated infrequently because developers do not encounter it during normal development. The further away documentation lives, the faster it drifts.

Good practice:

- Place a `README.md` in every package, module, or service directory that explains its purpose, inputs, outputs, and key design decisions
- Co-locate API reference documentation with the API source
- When moving or renaming a module, update documentation in the same commit
- Use relative links between documentation files to make co-location explicit and broken links detectable in CI

### DOC-PURPOSE — Every document has one clear purpose

A document should serve exactly one reader goal. The four fundamental documentation types — tutorials, how-to guides, reference material, and explanations — serve different purposes and should not be mixed.

Why it matters:

Mixed-purpose documents disorient readers because each documentation type requires a different reading posture. A reader following tutorial steps does not want to be interrupted by reference tables. Documents that serve multiple purposes satisfy none well, grow without bound, and become hard to maintain.

Good practice:

- Choose one documentation type before you start writing: tutorial, how-to, reference, or explanation
- State the document's purpose in the first sentence or opening section
- Link to related documents of other types rather than embedding their content
- Use the Diátaxis framework as a guide: each type has a distinct structure and tone

### DOC-MINIMAL — Write only what is necessary

Apply YAGNI to prose: do not document speculative features, rarely-used edge cases, or information the reader can derive from context. Every sentence has a maintenance cost.

Why it matters:

Over-documented systems create an illusion of completeness while hiding what matters under layers of noise. Readers learn to skim and miss critical information. Maintainers face an ever-growing surface area to keep accurate.

Good practice:

- Start with the minimum viable document: the one thing the reader must know to succeed
- Delete content that was true once but is no longer relevant
- Challenge every paragraph: what does the reader do differently after reading this?
- Apply the same code-review discipline to documentation PRs that you apply to code PRs

### DOC-AUDIENCE — Write for a specific, named audience

Every document should be written for a specific, identifiable reader with a known level of existing knowledge and a concrete goal.

Why it matters:

When a document lacks a defined audience, authors hedge — they over-explain concepts experts find patronizing, while glossing over assumptions that beginners need spelled out. The result is a document that neither audience finds useful.

Good practice:

- State the target audience and prerequisite knowledge in the first paragraph or "Before you begin" section
- If multiple audiences need the same topic, write separate documents or clearly labeled sections
- Use the audience's vocabulary, not internal jargon, unless you define it on first use
- Test the document with a representative reader from the intended audience before publishing

### DOC-ACCURACY — Keep documentation accurate and synchronized with the system

Documentation must accurately describe the current state of the system it documents.

Why it matters:

Inaccurate documentation causes wasted debugging time, incorrect integrations, and eroded trust. Once readers learn that docs are unreliable, they stop consulting them and resort to reading source code or asking colleagues.

Good practice:

- Include documentation updates in the definition of done for every code change that affects observable behavior
- Run documentation examples as part of the CI pipeline where possible (doc-tests, code samples in tests)
- Review documentation accuracy during API reviews and architecture reviews
- Mark known-stale sections explicitly rather than leaving readers to discover inaccuracies at runtime

### DOC-EXAMPLES — Illustrate with concrete, runnable examples

Abstract descriptions should be accompanied by concrete examples a reader can try immediately.

Why it matters:

Abstract descriptions leave readers uncertain whether they have understood correctly. A reader who must mentally simulate a system from prose alone is doing extra work that a good example would eliminate.

Good practice:

- Lead with the simplest possible working example before presenting variations or options
- Use realistic values in examples (real-looking URLs, plausible field names, actual command output)
- Show both input and output for any transformation or command
- Keep examples short: strip everything that does not contribute to illustrating the principle
- Test examples in CI where possible

### DOC-PROGRESSIVE-DISCLOSURE — Reveal complexity progressively

Present information in layers of increasing complexity: begin with the simplest case, then progressively reveal advanced options, edge cases, and deeper explanations.

Why it matters:

Front-loading complexity drives away readers who only need the simple case. Hiding advanced detail at the end frustrates experts who need it quickly.

Good practice:

- Open with the simplest possible working version of the thing being documented
- Structure headings to signal depth: "Basic usage" → "Configuration" → "Advanced options" → "Troubleshooting"
- Put the most common use case first; treat edge cases as separate sections
- Write a one-paragraph summary or TL;DR at the top of long documents
