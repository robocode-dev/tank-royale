# Prime

> **в›” PREREQUISITE вЂ” CHECK THIS BEFORE ANYTHING ELSE**
>

> - Explicit spec = `$ARGUMENTS` contains one or more `@`-prefixed tokens, or tokens matching `[A-Z][A-Z0-9]*(-[A-Z0-9]+)+` (bare principle IDs like `CODE-CS-DRY`)

You are activating a compact set of principles before working on a file or task. Follow these four phases exactly.

## Phase 1 вЂ” Parse Arguments and Guard

### 1.1 вЂ” Detect explicit mode

Scan `$ARGUMENTS` for explicit principle specs:

1. **`@<group>` tokens** вЂ” one or more words prefixed with `@` (e.g. `@ddd`, `@docs-as-code`). Extract all such tokens as the spec; the remaining tokens form the target context.
2. **Bare principle IDs** вЂ” tokens that match the pattern `[A-Z][A-Z0-9]*(-[A-Z0-9]+)+` (e.g. `CODE-CS-DRY`, `SOLID-SRP`). Extract all matching tokens as the spec; remaining tokens form the target context.
3. **No spec** вЂ” treat all of `$ARGUMENTS` as target context. This is normal mode.

If a spec was detected: set **explicit-mode: true**. Otherwise: **explicit-mode: false**.

### 1.3 вЂ” Detect artifact type

Determine the artifact type from the target context (remaining tokens after spec extraction, or the full `$ARGUMENTS` in normal mode). If the target context is empty or a description rather than a path, scan the current working directory.

Detect by reading `.principles-catalog/layers/artifact-types.yaml` and matching extensions, filenames, and path patterns.

Record the detected type: **`code`** | **`docs`** | **`config`** | **`infra`** | **`schema`** | **`pipeline`**

For **code** targets, also note any risk signals present: authentication, payments, PII, public API, concurrency, high-throughput.

## Phase 2 вЂ” Resolve Active Set

### Explicit mode

For each `@<group>` token: read `.principles-catalog/groups/<group>.yaml`. Add all IDs from its `principles:` list. Recursively process any `includes:` entries (detect cycles and abort). If a group file is not found, report a warning and continue.

For each bare principle ID token: add it directly to the active set.

Active set = union of all expanded IDs.

### Normal mode

Read all scout-generated files:
1. Glob `.github/instructions/*.instructions.md`
2. Glob `.claude/rules/*.md`
3. Filter to files containing the `<!-- generated by /dot-scout` marker
4. Parse every `- ID: Summary` line; the ID is everything before the first colon
5. Union all IDs в†’ **active set**

If no scout-generated files are found despite the install.cfg marker existing, stop and report:
> вљ пёЏ Scout marker found in install.cfg but no generated files detected. Re-run `/dot-scout` to regenerate them.

## Phase 3 вЂ” Select Top Principles

From the active set, select **5 to 10** principles most relevant to the current task. Hard cap: 10.

**Selection priority (fill from top until cap reached):**

1. **Risk-elevated** вЂ” any active IDs whose prefix is `OWASP-*`, `CODE-SEC-*`, `CODE-RL-*`, `INFRA-SEC-*`, or `SEC-ARCH-*`, AND which match a risk signal detected in Phase 1
2. **Universal** вЂ” `SIMPLE-DESIGN-REVEALS-INTENTION`, `CODE-CS-DRY`, `CODE-CS-KISS`, `CODE-CS-YAGNI`, `CODE-DX-NAMING`, `ARCH-DECISION-RECORDS`
3. **Most task-relevant** вЂ” use the artifact type and risk signals from Phase 1 to judge which remaining active IDs are most directly applicable to the work about to be done; prefer principles the developer is most likely to violate or forget for this specific task

In explicit mode, if the resolved set is в‰¤ 10: use all of them. If > 10: apply the same priority order to trim.

## Phase 4 вЂ” Output Active Rules

Output the selected principles as a compact block. For each principle, write a single line:

```
- ID: one imperative sentence summarising what to do
```

The imperative sentence must be в‰¤ 12 words, written in second-person present tense (e.g. "Give every piece of knowledge one authoritative home."). Derive it from the principle title and your knowledge of the principle вЂ” do not read any catalog files.

Format:

```
**Active rules:**
- CODE-CS-DRY: Give every piece of knowledge one authoritative home.
- SOLID-SRP: Give each module exactly one reason to change.
- CODE-CS-FAIL-FAST: Detect errors immediately and report them loudly.
...
```

End with this line, on its own:

> Apply these rules to all code you generate. Proceed.
