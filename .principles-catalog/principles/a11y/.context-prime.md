# .principles prime context — a11y
# Sections: Principle · Why it matters · Good practice

### A11Y-ALT-TEXT — Images must have meaningful alternative text

Every informative image must have an `alt` attribute that conveys the same information as the image. Decorative images must have `alt=""`. An absent `alt` is always a violation.

Why it matters:

Screen readers cannot interpret pixel content. Users who are blind or have images disabled rely entirely on alternative text. Missing or misleading `alt` text fails WCAG 2.1 SC 1.1.1 (Level A).

Good practice:

- Write `alt` text describing the *purpose or information*, not the visual appearance
- Do not start with "Image of" or "Picture of"
- For decorative images: `alt=""` and optionally `role="presentation"`
- For complex images (charts): include a full text description nearby or use `aria-describedby`
- Icon-only buttons must have `aria-label` describing the action

### A11Y-SEMANTIC-HTML — Use semantic HTML elements; do not build interactive controls from divs

Use the HTML element that most precisely describes the content's meaning. Do not replace `<button>`, `<a>`, `<nav>`, `<main>` with `<div>`/`<span>` plus ARIA. Native elements carry built-in keyboard behaviour and semantics that manual ARIA replication invariably misses.

Why it matters:

Assistive technologies rely on element type and ARIA landmark roles to model the page. A `<div>` with `onClick` is invisible as interactive to a screen reader, cannot be reached by keyboard, and provides no context about its purpose. Affects WCAG 2.1 SC 1.3.1 (Level A), SC 4.1.2 (Level A).

Good practice:

- `<button>` for actions; `<a href>` for navigation
- Heading levels `<h1>`–`<h6>` for document hierarchy, not for font sizing
- Landmark elements: `<main>`, `<header>`, `<footer>`, `<nav>`, `<aside>`
- Use ARIA only where native HTML is genuinely insufficient

### A11Y-KEYBOARD-NAVIGATION — Interactive elements must be fully keyboard-accessible

Every interactive element must be reachable via Tab, activatable via Enter/Space, and escapable via Escape. Mouse-only interactions are a hard barrier for keyboard and screen reader users.

Why it matters:

~7% of working-age adults have severe dexterity impairments. Screen reader users navigate entirely by keyboard. WCAG 2.1 SC 2.1.1 Keyboard (Level A) requires all functionality to be keyboard-accessible.

Good practice:

- Native `<button>` and `<a href>` are keyboard-accessible by default — prefer them
- If a div must be interactive: add `role`, `tabIndex={0}`, `onKeyDown` for Enter and Space
- Pair every `onMouseEnter`/`onMouseLeave` with `onFocus`/`onBlur`
- Never use `outline: none` without a visible custom focus indicator replacement
- Test by navigating the entire UI with keyboard only

### A11Y-COLOR-CONTRAST — Text must meet minimum contrast ratio against its background

Normal text (< 18pt / 14pt bold) needs 4.5:1 contrast; large text and UI component boundaries need 3:1. **Audit-scope: limited** — definitive evaluation requires rendered context; static analysis can only flag suspicious hardcoded values.

Why it matters:

~8% of males have some form of colour vision deficiency; contrast issues worsen for everyone in bright light, on low-end displays, and with age. Affects WCAG 2.1 SC 1.4.3 Contrast Minimum (Level AA).

Good practice:

- Use a contrast checker (WebAIM, browser DevTools accessibility panel, axe) to verify every text/background pair
- Centralised design tokens make systematic contrast review feasible
- Never rely on colour alone to convey information — add a secondary indicator (icon, label, pattern, underline)
- Ensure focus indicator contrast meets 3:1 (SC 1.4.11)
