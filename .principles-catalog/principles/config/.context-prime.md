### CONFIG-NO-HARDCODED-SECRETS â€” Never hardcode secrets in configuration files

Secrets â€” passwords, API keys, tokens, database credentials, private keys â€” must never appear in configuration files committed to version control.

Why it matters:

A hardcoded secret in a committed config file is effectively a public credential. Once committed, it exists in git history even after deletion. Breaches from leaked credentials are among the most common causes of data breaches.

Good practice:

- Use environment variable references: `password: ${DB_PASSWORD}` or `password: {{ env "DB_PASSWORD" }}`
- Store only `.env.example` (with placeholder values) in VCS; keep `.env` in `.gitignore`
- Use secrets management integrations: Vault, AWS Secrets Manager, Kubernetes secrets via `secretKeyRef`
- Rotate any secret that has been committed, even temporarily â€” git history is permanent

### CONFIG-SCHEMA-VALIDATION â€” Validate configuration against a schema at startup

Configuration should be validated against a schema at application startup, causing an immediate descriptive failure if values are missing, malformed, or out of range.

Why it matters:

Configuration errors are one of the most common causes of production incidents. When missing or malformed config is not caught at startup, the application fails later â€” often at the worst possible moment.

Good practice:

- Use a typed configuration schema: Pydantic `BaseSettings`, Spring Boot `@ConfigurationProperties`, Go `envconfig`, .NET `IOptions<T>`
- Validate at the composition root â€” before any application logic runs
- Fail with a descriptive error naming the missing key and its expected format
- Separate required from optional config in the schema; optional values should have explicit documented defaults

### CONFIG-SCHEMA-FIRST â€” Declare a schema for every configuration file

Before writing configuration values, define a schema that declares every expected key, its type, constraints, and whether it is required or optional.

Why it matters:

Without a declared schema, configuration files drift: keys accumulate without documentation, types change silently, and the only truth about what the application expects is buried in application code.

Good practice:

- Add `$schema: "path/to/schema.json"` or `# yaml-language-server: $schema=...` to YAML files so editors validate on save
- Define application configuration as a typed schema first: Pydantic, Spring Boot `@ConfigurationProperties`, Go struct, Zod object
- Store schema alongside config in VCS; update both in the same commit
- Validate config against schema in CI using `ajv`, `jsonschema`, `check-jsonschema`, or equivalent

### CONFIG-EXPLICIT-OVER-CONVENTIONAL â€” Prefer explicit keys over magic conventions

Configuration keys and their effects must be explicit and self-evident â€” not dependent on knowing a framework's undocumented defaults or magic conventions.

Why it matters:

Convention-based configuration creates invisible coupling between config files and framework internals. When defaults change between framework versions, or when a new team member inherits the system, implicit conventions become subtle bugs that are hard to detect.

Good practice:

- Name every key that influences behaviour, even when the value matches the default; this makes intent visible and diffs meaningful
- Document what each key does and what value the application uses when it is absent
- Use schema `default` declarations rather than burying fallback logic in application code
- Prefer `feature_x_enabled: false` over omitting the key entirely
- Keep a `.env.example` or `config.example.yaml` that lists every supported key with a description

### CONFIG-ENVIRONMENT-PARITY â€” Keep config structure identical across environments

Config schema and structure must be identical across all environments; only values differ; code never branches on environment name.

Why it matters:

When dev and production config schemas diverge, a missing key in production is only discovered at deployment. Environment-specific branching hides prod-only code paths that never run in development, making the most critical paths the least tested.

Good practice:

- Use a single config schema for all environments; populate it with environment-specific values via environment variables or per-environment files
- Replace `if env == "production": enable_feature_x()` with `feature_x_enabled: ${FEATURE_X_ENABLED:-false}` in config
- Test with production-like configuration values in CI to surface schema drift early
- Validate that all environments use the same config schema version in CD pipelines

### CONFIG-EXPLICIT-DEFAULTS â€” Declare and document defaults for every optional key

Every optional configuration key must have an explicit, documented default value declared in the schema â€” not inferred from application fallback logic buried in code.

Why it matters:

Implicit defaults are hidden assumptions. A connection timeout defaulting to 30 seconds or a pool size defaulting to 10 are invisible until a production incident surfaces them. When defaults live in application code rather than configuration schemas, they are hard to find and impossible to validate without running the application.

Good practice:

- Declare defaults in schema: Pydantic `Field(default=30)`, JSON Schema `"default": 30`, Spring Boot `@Value("${timeout:30}")`
- Document each default inline: what it is, why it was chosen, and when it should be overridden
- Make production-unsafe defaults obvious: `log_level: DEBUG  # override to INFO or WARNING in production`
- Use `.env.example` or `values.example.yaml` showing recommended production values next to development defaults

### CONFIG-CHANGE-TRACEABILITY â€” All configuration changes must be versioned, reviewed, and auditable

Every change to a configuration value must be traceable: who changed it, what changed, when, and why.

Why it matters:

Configuration changes are a leading cause of production incidents. Without traceability, diagnosing "what changed before the outage" requires guesswork. Without review, a single misconfiguration can take down a service.

Good practice:

- Commit all configuration files to version control; require pull request approval before merging config changes
- Use GitOps workflows (ArgoCD, Flux) to deploy config from VCS, ensuring running config matches declared config
- Enable audit logging in secrets management systems: Vault audit devices, AWS CloudTrail, Azure Policy
- Define and document break-glass procedures for emergency changes; require a follow-up PR to reconcile VCS within 24 hours

### CONFIG-MINIMAL-SURFACE â€” Expose only the configuration the application reads

Config files should contain only the keys the application actively reads; unused, deprecated, and commented-out keys must be removed.

Why it matters:

Dead configuration keys create confusion, hide security exposure, and make audits harder. OWASP A05 identifies unnecessary features and unremediated default configuration as primary misconfiguration sources.

Good practice:

- Remove config keys when the feature they configure is removed; treat this as part of the same PR
- Periodically cross-reference config keys against application source to confirm each is still read
- Document deprecated keys with a removal target version; enforce removal in the schema deprecation lifecycle
- Start from a minimal config and add keys only when the application explicitly needs them
