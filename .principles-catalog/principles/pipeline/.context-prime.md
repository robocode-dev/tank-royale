# .principles prime context — pipeline
# Sections: Principle · Why it matters · Good practice

### PIPELINE-MINIMAL-PERMISSIONS — Run pipeline jobs with minimal permissions

Every pipeline job should run with the minimum permissions required to complete its specific task.

Why it matters:

Over-permissioned pipelines are a high-value attack surface. A compromised build step with write access to all production environments can escalate to a full production breach. Supply chain attacks increasingly target CI/CD because pipelines often run with elevated credentials.

Good practice:

- Declare explicit minimum permissions per job in CI config
- In GitHub Actions: `permissions: contents: read` for build jobs; `permissions: id-token: write` only for OIDC-based deploy jobs
- Use short-lived credentials via OIDC federation rather than long-lived static secrets
- Create separate service accounts per pipeline stage; never share credentials across build and deploy

### PIPELINE-NO-SECRETS-IN-LOGS — Never expose secrets in pipeline logs

Secrets, tokens, passwords, and credentials must never appear in CI/CD pipeline logs or job output.

Why it matters:

Pipeline logs are often accessible to the whole engineering team, stored for extended periods, and may be exported to log aggregators. A single `echo $SECRET` can leak a credential into a persistent, broadly-accessible record.

Good practice:

- Always use masked/secret variables in CI/CD platforms (GitHub Actions secrets, GitLab masked variables)
- Never use `set -x` in script sections that handle secrets
- Pass secrets via stdin or a credentials file rather than command-line arguments
- Audit pipeline logs periodically with secret scanning tools

### PIPELINE-REPRODUCIBLE-BUILDS — Ensure builds produce identical outputs from identical inputs

A pipeline build is reproducible when the same source code, dependency set, and build toolchain produce a bit-for-bit identical artifact every time, regardless of when or where the build runs.

Why it matters:

Non-reproducible builds undermine every downstream guarantee. If rebuilding from the same commit produces a different binary, you cannot verify that what was tested is what ships. Attackers can exploit non-determinism to inject code that only appears in certain build environments. Reproducibility is a prerequisite for SLSA Level 3 provenance.

Good practice:

- Use `npm ci`, `yarn install --frozen-lockfile`, or `pnpm install --frozen-lockfile` — fail if the lockfile is out of sync
- Pin build tool images to a digest rather than a mutable tag
- Vendor dependencies or use a private artifact proxy so builds do not depend on external registries at build time
- Avoid embedding timestamps or non-deterministic values in artifacts; use `SOURCE_DATE_EPOCH` where supported
- Commit all lockfiles to version control; treat lockfile drift as a build failure

### PIPELINE-ENVIRONMENT-ISOLATION — Pipeline stages must not share mutable state

Each pipeline stage must execute in an isolated context with its own workspace, credentials, and environment variables. A failing or compromised stage must not be able to corrupt the workspace of another stage.

Why it matters:

Shared mutable state between pipeline stages is a source of flaky builds, phantom test passes, and security escalation. A test stage that writes to a shared directory can cause a subsequent deploy stage to ship untested files. Shared credentials mean a compromised build step can access production secrets intended only for the deploy step.

Good practice:

- Use explicit artifact passing between stages: one stage produces a named artifact, the next stage downloads it by name
- Scope credentials per stage: build steps receive read-only credentials, deploy steps receive deploy credentials
- Run each stage in a fresh container or clean workspace
- Use environment protection rules to restrict which stages can access which secrets

### PIPELINE-FAIL-FAST-PIPELINE — Abort the pipeline on the first meaningful failure

When a pipeline stage fails, subsequent dependent stages must not execute. The pipeline should abort as early as possible and report the failure clearly.

Why it matters:

A pipeline that continues after failure wastes compute resources, delays feedback, and produces confusing results. Developers see green deploy stages downstream of a red test stage and may wrongly assume success. In the worst case, a pipeline that deploys despite test failures puts broken code into production.

Good practice:

- Define explicit stage dependencies so deploy stages only run after all test stages succeed
- Reserve `continue-on-error` for genuinely non-blocking advisory checks (optional linting, coverage reporting)
- Use `set -euo pipefail` at the top of bash scripts in pipeline steps
- Configure the pipeline to report the first failure prominently

### PIPELINE-DEPLOYMENT-GATES — Require explicit approval or conditions before production deployment

Deployment to production must pass through an explicit gate — a manual approval, an automated policy check, or both — enforced by the pipeline platform.

Why it matters:

Without deployment gates, any pipeline run that reaches the deploy stage can push directly to production. Gates provide a deliberate checkpoint that separates "code that passed automated checks" from "code approved for production." This is critical in regulated industries where audit trails of production approvals are required.

Good practice:

- Use platform-native environment protection rules: GitHub Environments with `required_reviewers`, GitLab protected environments
- Restrict production deployment triggers to specific branches (`main`, `release/*`)
- Require at least one human approval for production deployments; automate staging deployments
- Combine manual approval with automated policy checks: verify the artifact passed all test stages and security scans before allowing deployment
