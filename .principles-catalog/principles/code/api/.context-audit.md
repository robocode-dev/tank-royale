# .principles audit context — code/api
# Sections: Principle · Violations to detect

### CODE-API-API-VERSIONING — Version APIs explicitly and define a policy for breaking changes

Every API with external consumers must carry an explicit version identifier. Breaking changes require a new version with a documented deprecation period. Additive changes do not require a new version.

Violations to detect:

- Public API endpoints with no version identifier in the URI, header, or media type
- Breaking changes deployed to an existing version without a new version (removing fields, changing types, making optional params required)
- No deprecation policy — no timeline, no migration guide, no `Sunset` header on deprecated endpoints
- Deprecated endpoints missing `Deprecation` or `Sunset` response headers
- Inconsistent versioning strategies mixed across the same API

### CODE-API-BACKWARD-COMPATIBILITY — Support backward compatibility — never break existing clients

Once an API is published and clients depend on it, changes must be backward compatible. Adding new fields, endpoints, or optional parameters is safe; removing fields, renaming parameters, changing the type of a return value, or altering the meaning of existing behavior is not.

Violations to detect:

- Removing or renaming fields in response bodies without versioning
- Changing the type of an existing field (e.g., string to integer, single value to array)
- Making a previously optional request parameter required
- Changing the meaning or behavior of an existing endpoint without a version bump
- Removing or renaming endpoints that clients may depend on
- Altering error response formats or status codes for existing error conditions

### CODE-API-GRPC-PROTOBUF — Define gRPC service contracts in .proto files; evolve them without breaking existing clients

gRPC service contracts must be defined in `.proto` files as the authoritative contract. Field numbers are permanent — never reuse a number for a different field. Use `reserved` to retire fields.

Violations to detect:

- Removing a field from a `.proto` message without adding its number and name to a `reserved` statement
- Reusing a field number for a new field after a previous field at that number was deleted
- Changing the type of an existing field (e.g., `string` to `int32`, `optional` to `repeated`)
- Renaming an RPC method or message type without a package version bump
- `.proto` files that have drifted from the running implementation

### CODE-API-CONDITIONAL-REQUESTS — Support conditional requests for safe concurrent writes and cache revalidation

Resources that may be updated concurrently must support conditional write semantics using ETag / If-Match. GET endpoints should honour If-None-Match to return 304 Not Modified when the client's cached copy is still current.

Violations to detect:

- PUT or PATCH endpoints that apply updates to concurrently editable resources without checking `If-Match`
- GET endpoints that return mutable resources without an `ETag` or `Last-Modified` header
- `If-None-Match` requests that receive a full 200 response instead of 304 when the ETag matches
- `If-Match` failures that return 200 (success) instead of 412 (precondition failed)
- `If-Match: *` not handled — this wildcard must succeed only if the resource exists

### CODE-API-CONTENT-NEGOTIATION — Honour the Accept header and respond with the negotiated content type

APIs must inspect the client's Accept header and respond with a representation in one of the requested media types. Always set a precise Content-Type header on the response.

Violations to detect:

- Endpoints that always respond with `application/json` regardless of the client's `Accept` header
- Missing or incorrect `Content-Type` header on responses
- No handling for `Accept: */*` — the server must select its default type and declare it in `Content-Type`
- Returning 500 or 400 instead of 406 when the client requests an unsupported media type
- Request bodies parsed without checking `Content-Type`, silently accepting arbitrary or malformed formats

### CODE-API-HATEOAS — Make APIs self-descriptive using hypermedia (HATEOAS)

A REST API should drive application state through hypermedia — each response should include links and controls that tell the client what it can do next.

Violations to detect:

- Clients that construct URIs by string concatenation rather than following links from responses
- API responses that return only data with no navigational links or action controls
- Tight coupling between client routing logic and server URL structure
- API documentation that lists endpoints as a flat table of URL patterns with no mention of how responses link to one another

### CODE-API-HTTP-CACHING — Set Cache-Control and validation headers on cacheable responses

Every GET response must carry explicit Cache-Control directives. Cacheable resources should include ETag or Last-Modified headers to support conditional revalidation.

Violations to detect:

- GET endpoints that return responses with no `Cache-Control` header
- Sensitive or personalised responses without `Cache-Control: private` or `no-store`
- Cacheable resources with no `ETag` or `Last-Modified` header
- POST, PUT, PATCH, or DELETE responses carrying a `Cache-Control` header that permits caching
- Overly long `max-age` values (days or weeks) on frequently-changing resources

### CODE-API-HTTP-STATUS-CODES — Use proper HTTP status codes to communicate outcomes

APIs must use HTTP status codes correctly: 2xx for success, 4xx for client errors, 5xx for server errors. Each specific code carries precise semantics.

Violations to detect:

- Returning 200 OK for error conditions, with the actual error buried in the response body
- Using 500 Internal Server Error for client mistakes that should be 400 or 422
- Returning 404 for authorization failures that should be 403 or 401
- Using 400 Bad Request as a catch-all instead of more specific codes (409, 422, 429)
- Never returning 201 Created for successful resource creation

### CODE-API-PAGINATION — Paginate all collection resources; never return unbounded results

Collection endpoints must never return an unbounded number of items. Implement cursor-based or offset-based pagination with navigation links, and enforce a maximum page size server-side.

Violations to detect:

- Collection endpoints that return a plain array with no pagination metadata or navigation links
- `getAll()`, `findAll()`, or `SELECT *` queries against large tables without a `LIMIT` clause
- A `pageSize` or `limit` parameter accepted but not capped server-side
- No `next` link or cursor in the response, leaving clients unable to detect incomplete results

### CODE-API-PROBLEM-DETAILS — Return structured error responses using the RFC 7807 Problem Details format

Error responses must be machine-readable and consistent, using RFC 7807 Problem Details: `type`, `title`, `status`, `detail`, and optionally `instance`. Use `Content-Type: application/problem+json`.

Violations to detect:

- Error responses using ad-hoc fields (`message`, `error`, `errorCode`) instead of `type`/`title`/`status`/`detail`
- Stack traces, exception class names, or internal identifiers exposed in error response bodies
- Inconsistent error shapes across endpoints
- 4xx and 5xx responses with an empty body or an HTML error page
- Missing `status` field in the error body

### CODE-API-RATE-LIMITING — Enforce rate limits and signal capacity constraints with standard headers

APIs must enforce rate limits and return 429 Too Many Requests with a Retry-After header when a limit is exceeded. Advertise quota state via RateLimit-* headers.

Violations to detect:

- API endpoints without any rate-limiting middleware or filter in the request pipeline
- 429 responses that omit the `Retry-After` header
- Rate limit errors returned as 500 or 400 instead of 429
- No `RateLimit-*` or `X-RateLimit-*` headers on responses

### CODE-API-RESOURCE-NOUNS — Design resources around nouns, not verbs

REST APIs expose resources identified by URIs. Resource URIs should be nouns representing domain entities; the HTTP method supplies the verb.

Violations to detect:

- URIs containing action verbs: `/getUser`, `/createOrder`, `/deleteItem`, `/updateProfile`
- Endpoints that duplicate the HTTP method in the path: `POST /api/createUser` instead of `POST /api/users`
- Inconsistent resource naming — mixing plural and singular forms
- Deeply nested verb paths where a state-transition sub-resource would be more RESTful

### CODE-API-STANDARD-HTTP-METHODS — Use standard HTTP methods with their defined semantics

Each HTTP method carries a defined contract: GET is safe (no side effects), PUT and DELETE are idempotent, POST is neither safe nor idempotent. APIs must honour these semantics.

Violations to detect:

- GET endpoints that create, update, or delete resources
- PUT endpoints that are not idempotent — calling them twice produces different results
- PATCH requests that send a full resource representation instead of a partial update
- DELETE endpoints that are not idempotent
- POST used where PUT would be semantically correct (full resource replacement)
