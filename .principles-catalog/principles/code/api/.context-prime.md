# .principles prime context — code/api
# Sections: Principle · Why it matters · Good practice

### CODE-API-API-VERSIONING — Version APIs explicitly and define a policy for breaking changes

Every API with external consumers must carry an explicit version identifier. Breaking changes require a new version. Additive changes (new optional fields, new endpoints) do not. Maintain the old version for a documented deprecation period.

Why it matters:

Without versioning, any breaking change forces all consumers to update simultaneously or break. A clear versioning policy gives consumers a predictable migration window and gives maintainers freedom to evolve the API without holding it hostage to the slowest consumer.

Good practice:

- Use URI path versioning (`/v1/orders`) as the default — visible, cacheable, unambiguous
- Define a deprecation policy before publishing v1: minimum notice period, overlap window, migration documentation
- Set `Deprecation: true` and `Sunset: <HTTP-date>` headers on deprecated endpoints
- Keep simultaneously supported versions small (typically two: current and previous)

### CODE-API-BACKWARD-COMPATIBILITY — Support backward compatibility — never break existing clients

Once an API is published and clients depend on it, changes must be backward compatible. Adding new fields, endpoints, or optional parameters is safe; removing, renaming, or changing existing behavior is not. When a breaking change is unavoidable, introduce a new API version.

Why it matters:

APIs are shared contracts. Unlike internal code, you do not control who calls your API or when they update. A breaking change that seems minor — removing an unused field, changing a date format — can cause silent data loss or outright failures in client applications. The cost multiplies with the number of clients and the difficulty of coordinating updates.

Good practice:

- Treat published API fields and endpoints as immutable contracts — add, never remove or rename
- Use additive changes: new optional fields, new endpoints, new optional query parameters
- When breaking changes are necessary, use explicit versioning (`/v2/`) with a deprecation policy and migration window
- Use contract testing (consumer-driven contracts) to detect accidental breaking changes before release

### CODE-API-GRPC-PROTOBUF — Define gRPC service contracts in .proto files; evolve them without breaking existing clients

gRPC contracts must live in `.proto` files as the authoritative source of truth. Field numbers are permanent binary identifiers — once published, a number must never be reused for a different field. Use `reserved` to retire fields.

Why it matters:

Protobuf's binary encoding relies on field numbers, not names. Reusing a number silently produces wrong data for unupdated clients rather than an error — incompatibilities are invisible until they corrupt a message. The `.proto` file is the only human-readable contract; if it drifts from the implementation, consumers cannot trust it.

Good practice:

- Treat the `.proto` file as source of truth — generate stubs and SDKs from it, never the reverse
- Reserve deleted field numbers and names immediately: `reserved 3, 5; reserved "old_field_name";`
- Never change the type or cardinality of an existing field; add a new field with a new number instead
- Use package versioning for breaking changes: `package myservice.v1;` → `package myservice.v2;`
- Run `buf lint` and `buf breaking` in CI to catch regressions automatically

### CODE-API-CONDITIONAL-REQUESTS — Support conditional requests for safe concurrent writes and cache revalidation

Resources that may be updated concurrently must support conditional write semantics using ETag / If-Match. A client reads a resource, receives its ETag, and includes If-Match on any subsequent write. If the resource has changed, the server returns 412 instead of overwriting. GET endpoints should honour If-None-Match to return 304 when the client's cached copy is current.

Why it matters:

Without conditional requests, concurrent clients performing read-modify-write cycles silently overwrite each other's changes — the last writer always wins and earlier writes are lost with no error. Conditional GETs allow clients and caches to avoid re-downloading unchanged resources, reducing bandwidth and latency.

Good practice:

- Return an `ETag` header on every GET response for mutable resources; the value should be a hash or version token
- Require `If-Match` on PUT and PATCH for concurrently editable resources; return 412 if the condition fails
- Honour `If-None-Match` on GET; return 304 with no body if the ETag matches
- Use strong ETags (`"abc123"`) by default; weak ETags (`W/"abc123"`) only for semantically equivalent representations

### CODE-API-CONTENT-NEGOTIATION — Honour the Accept header and respond with the negotiated content type

APIs must inspect the client's Accept header and respond with a representation in one of the requested media types. If no supported type can be produced, return 406. Always set a precise Content-Type header on the response.

Why it matters:

Hardcoding `application/json` prevents the API from serving clients with different requirements and leaks representation decisions into client code, tightly coupling clients to the server's current choice. Content negotiation is a built-in HTTP mechanism for decoupling clients from representation details without changing the API surface.

Good practice:

- Parse the `Accept` header and select the best match using q-value weighted preference
- Default to `application/json` when the client sends `Accept: */*` or omits the header; always declare `Content-Type` explicitly
- Return 406 Not Acceptable with a list of supported media types when no match is possible
- Support `application/problem+json` for error responses alongside the normal content types
- Declare supported media types per endpoint in the OpenAPI specification

### CODE-API-HATEOAS — Make APIs self-descriptive using hypermedia (HATEOAS)

A REST API should drive application state through hypermedia — each response includes links and controls that tell the client what it can do next. The client should not need to hardcode URI structures or out-of-band knowledge to navigate the API.

Why it matters:

When clients hardcode URI templates, the server cannot evolve its URI structure without breaking them. Hypermedia decouples clients from server implementation details, allowing the server to change URIs, add capabilities, or restructure resources — well-behaved clients discover these changes through links in responses.

Good practice:

- Include `_links` or equivalent hypermedia controls in every resource representation
- Use established hypermedia formats (HAL, JSON:API, Siren, or Hydra) rather than inventing a custom link structure
- Return links that reflect the current resource state — only advertise actions the client is authorised and able to perform
- Start clients from a single well-known entry-point URL; all other URIs are discovered from responses

### CODE-API-HTTP-CACHING — Set Cache-Control and validation headers on cacheable responses

Every GET response must carry explicit Cache-Control directives. Cacheable resources should include ETag or Last-Modified headers. Sensitive resources must be explicitly marked no-store. Never rely on default intermediary caching behaviour.

Why it matters:

Caching is the primary mechanism for reducing latency and origin load. CDNs and proxies make decisions based on Cache-Control. Omitting headers hands the decision to each intermediary's defaults — some will cache everything (including private data), others nothing. Incorrect caching can serve stale data, expose private information, or re-fetch unchanged resources on every request.

Good practice:

- Rarely-changing public resources: `Cache-Control: public, max-age=3600, stale-while-revalidate=60`
- Private user-specific data: `Cache-Control: private, no-store`
- Real-time data: `Cache-Control: no-cache, no-store`
- Include `ETag` on cacheable resources; enables `304 Not Modified` and avoids re-transmitting unchanged data
- Use `Vary: Accept-Encoding, Accept` when responses differ by negotiated content type or encoding

### CODE-API-HTTP-STATUS-CODES — Use proper HTTP status codes to communicate outcomes

Use HTTP status codes correctly: 2xx for success, 3xx for redirection, 4xx for client errors, 5xx for server errors. Each specific code carries precise semantics that clients, caches, and infrastructure rely on.

Why it matters:

Clients, proxies, caches, monitoring tools, and load balancers all interpret status codes to make decisions. A cache stores a 200 but not a 500. A retry library retries a 503 but not a 400. Monitoring dashboards flag 5xx spikes as incidents. Returning 200 for errors breaks all of this infrastructure and forces clients to parse bodies to determine what actually happened.

Good practice:

- 200 OK for retrieval/general success, 201 Created when a resource is created, 204 No Content for no-body success
- 400 for malformed syntax, 401 for missing/invalid auth, 403 for insufficient permissions, 404 for not found, 409 for conflicts, 422 for semantic validation failures, 429 for rate limiting
- 500 only for genuine unexpected server failures — never for input validation or business rule violations
- Always include a structured error body with a machine-readable code, human-readable message, and field pointer where applicable

### CODE-API-PAGINATION — Paginate all collection resources; never return unbounded results

Collection endpoints must never return an unbounded number of items. Implement cursor-based or offset-based pagination and include navigation links. Enforce a maximum page size server-side.

Why it matters:

An unbounded collection response becomes a reliability and performance hazard as data grows. A table with millions of rows causes out-of-memory errors, timeouts, and database strain. Cursor-based pagination avoids the "missing row" and "duplicate row" anomalies caused by insertions during offset-based traversal.

Good practice:

- Return a consistent envelope: `{ "data": [...], "meta": { "total": 1500, "pageSize": 20 }, "links": { "next": "...", "prev": "..." } }`
- Enforce a hard maximum page size server-side regardless of what the client requests
- For cursor pagination, make the cursor opaque (base64-encode internal state); never expose raw offsets as cursors on mutable datasets
- Return `data: []` rather than 404 when a page is valid but empty
- Include `total` in offset pagination for UI page-count components; omit it in cursor pagination

### CODE-API-PROBLEM-DETAILS — Return structured error responses using the RFC 7807 Problem Details format

Error responses must be machine-readable and consistent using RFC 7807 Problem Details: type, title, status, detail, and optionally instance. Use Content-Type: application/problem+json. Never expose stack traces or internal details.

Why it matters:

Inconsistent error formats force consumers to write custom parsing for every error shape across every endpoint. A standard format lets clients handle all errors with one code path and lets monitoring tools parse and categorise errors automatically. Exposing exception types and stack traces also leaks information that aids attackers.

Good practice:

- Return `Content-Type: application/problem+json` for all error responses
- Minimum required fields: `type`, `title`, `status`, `detail`
- Use a stable, documented URI as `type` that identifies the class of error
- Handle all unhandled exceptions in a global error handler that maps them to Problem Details responses
- Log the `instance` URI server-side to correlate client-reported errors with server traces

### CODE-API-RATE-LIMITING — Enforce rate limits and signal capacity constraints with standard headers

APIs must enforce rate limits and return 429 Too Many Requests with a Retry-After header when a limit is exceeded. Expose quota state via RateLimit-* headers so clients can self-throttle.

Why it matters:

An API without rate limiting is vulnerable to accidental overload and deliberate abuse. A single runaway consumer can degrade or bring down the service for everyone. Standard headers allow well-behaved clients to pace themselves, reducing retries and dropped requests.

Good practice:

- Apply rate limiting as middleware at the API gateway or framework filter level — not per endpoint
- Return `Retry-After` alongside every 429 response
- Expose quota headers on responses: `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`
- Distinguish user-level, application-level, and IP-level limits; identify which limit was exceeded in the 429 body
- Document rate limit policies in the API reference

### CODE-API-RESOURCE-NOUNS — Design resources around nouns, not verbs

REST APIs expose resources identified by URIs. Resource URIs should be nouns representing domain entities; the HTTP method supplies the verb. This separation is central to the uniform interface constraint.

Why it matters:

Verb-based URIs drift toward RPC style, leading to an explosion of endpoints that are difficult to discover, cache, and document. Noun-based resources map naturally to CRUD operations through HTTP methods, producing a smaller, more consistent API surface that intermediaries can reason about.

Good practice:

- Use plural nouns for collections (`/orders`), resource ID for individuals (`/orders/42`)
- Let HTTP methods express the action: GET (retrieve), POST (create), PUT (replace), PATCH (partial update), DELETE (remove)
- For operations that do not map cleanly to CRUD, model the action as a sub-resource (e.g., `POST /orders/42/cancellations`)

### CODE-API-STANDARD-HTTP-METHODS — Use standard HTTP methods with their defined semantics

Each HTTP method carries a defined contract: GET is safe (no side effects), PUT and DELETE are idempotent, POST is neither safe nor idempotent. APIs must honour these semantics so that clients, caches, and intermediaries can rely on them.

Why it matters:

The uniform interface is what makes the web scalable. Intermediaries make decisions based on method semantics — caches replay GET requests safely, clients retry PUT after a network failure. When an API misuses methods, these assumptions break, leading to data corruption, cache poisoning, and unreliable behaviour under failure conditions.

Good practice:

- GET for retrieval only — never modify server state on a GET
- PUT for full resource replacement; ensure idempotency — the same PUT with the same body always produces the same state
- POST for creating resources when the server assigns the identifier, or for non-idempotent operations
- DELETE for resource removal; make it idempotent — deleting an already-deleted resource returns 404 or 204, not an error
- PATCH for partial updates; document whether your PATCH implementation is idempotent
