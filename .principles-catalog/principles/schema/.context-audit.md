# .principles audit context — schema
# Sections: Principle · Violations to detect

### SCHEMA-SELF-DESCRIBING — Make schemas self-describing

Every field, type, enum value, and RPC method in a schema should carry enough description, naming clarity, and structural context that a consumer can understand its purpose without consulting external documentation.

Violations to detect:

- Fields named `data`, `value`, `info`, `obj`, `result`, or other generic names without descriptions
- Enum values with no indication of what each variant means (e.g., `STATUS_1`, `STATUS_2`)
- Protobuf fields with no comment blocks explaining units, constraints, or expected format
- OpenAPI schema properties with no `description` field
- SQL columns with abbreviations that require tribal knowledge to decode
- RPC method names that are verbs without object context (`Process`, `Handle`, `Execute`)

### SCHEMA-FIELD-OPTIONALITY — New fields must be optional; never add required fields to published schemas

Every new field added to a published schema must be optional and carry a sensible default value. Adding a required field to a schema that already has consumers is a breaking change — existing producers will generate data that fails validation, and existing consumers will reject data lacking the new field.

Violations to detect:

- A `required` annotation on a field not present in the schema's initial published version
- Protobuf fields without `optional` keyword in proto3 that carry meaning distinct from the zero value
- OpenAPI `required` array including a field that did not exist in the previous published version
- Avro record fields without a `default` value
- JSON Schema `required` list containing newly added properties
- GraphQL non-null (`!`) fields added to existing types after initial publication

### SCHEMA-NO-POLYMORPHIC-BLOBS — Do not use untyped blobs that bypass schema validation

Schema fields must not use untyped, opaque containers — `Any`, `Object`, `bytes`, `JSON` columns, `additionalProperties: true` — to carry structured data that should have its own schema. Every piece of data that crosses a boundary must be described by an explicit schema.

Violations to detect:

- `google.protobuf.Any` or `google.protobuf.Struct` used as a field type in Protobuf messages
- `bytes` fields carrying serialized structured data rather than genuinely opaque binary
- OpenAPI schemas using `type: object` without `properties`
- OpenAPI schemas with `additionalProperties: true` on top-level request or response body
- JSON Schema using `{}` (empty schema) or `true` as a property schema
- SQL columns of type `JSON`, `JSONB`, `TEXT`, or `BLOB` storing structured domain data without a validated internal schema
- GraphQL `JSON` or `Any` scalar types used to pass structured data

### SCHEMA-ENUM-EVOLUTION — Enums must be safe to extend; consumers must handle unknown values

Every enum in a published schema must include an explicit unknown or unspecified sentinel value, and every consumer must handle unrecognised values without crashing. Adding a new variant to an enum is routine schema evolution; if any consumer treats an unknown variant as an error, enum extension becomes a breaking change.

Violations to detect:

- Protobuf enums whose first value (field number 0) is not an explicit `UNSPECIFIED` or `UNKNOWN` sentinel
- OpenAPI enums without an explicit `UNKNOWN` / `OTHER` variant or non-exhaustive documentation
- `switch` / `match` / `case` statements on enum values without a `default` / `_` / `else` branch
- Client-generated code that throws on enum values not in the spec's `enum` list
- Avro enums without a `default` symbol
