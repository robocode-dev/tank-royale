# .principles prime context — schema
# Sections: Principle · Why it matters · Good practice

### SCHEMA-SELF-DESCRIBING — Make schemas self-describing

Every field, type, enum value, and RPC method in a schema should carry enough description, naming clarity, and structural context that a consumer can understand its purpose without consulting external documentation.

Why it matters:

Schemas are consumed by developers who did not write them, often months or years after creation. A schema without descriptions forces every consumer to reverse-engineer intent from field names alone, leading to misuse and integration bugs.

Good practice:

- Every field in a public schema must have a description explaining what it represents, its units, and any constraints
- Enum variants should have names that read as domain concepts: `ORDER_STATUS_PENDING` not `STATUS_1`
- Use consistent naming conventions throughout a schema
- Include examples in OpenAPI schemas (`example:` property) to make the format concrete
- In Protobuf, use field comments to document backward-compatibility constraints

### SCHEMA-FIELD-OPTIONALITY — New fields must be optional; never add required fields to published schemas

Every new field added to a published schema must be optional and carry a sensible default value. Adding a required field to a schema that already has consumers is a breaking change — existing producers will generate messages that fail validation, and existing consumers will reject data lacking the new field.

Why it matters:

In a distributed system, producers and consumers are upgraded independently. A new required field creates a deployment-order dependency: every producer must be updated before any consumer running the new schema version can accept messages. Optional fields with defaults avoid this — old producers omit the field, new consumers use the default, and the system functions during the rolling upgrade window.

Good practice:

- Make every new field optional with an explicit default that preserves existing behaviour when the field is absent
- In Protobuf (proto3), document the semantic meaning of the zero value for every field — it is the implicit default
- In Avro, always specify a `default` for every field; without one, schema resolution fails
- In OpenAPI, do not add new property names to the `required` array of an existing schema after publication
- In GraphQL, add new fields as nullable; non-null (`!`) is only safe on fields present since initial publication
- Enforce schema compatibility checks in CI using a schema registry or diff tool

### SCHEMA-NO-POLYMORPHIC-BLOBS — Do not use untyped blobs that bypass schema validation

Schema fields must not use untyped, opaque containers — `Any`, `Object`, `bytes`, `JSON` columns, `additionalProperties: true`, or equivalent — to carry structured data that should have its own schema. Every piece of data crossing a service boundary must be described by an explicit schema so it can be validated, evolved, and understood.

Why it matters:

An untyped blob inside a schema is a hole in the type system. It cannot be validated by schema tools, cannot be evolved with compatibility checks, and cannot be understood by consumers without consulting the producer's source code. Bugs caused by untyped blobs are discovered at runtime because no compile-time or schema-validation-time check can catch them.

Good practice:

- Define explicit message types or schema objects for every structured payload — never use `Any`, `Object`, or `bytes` as a shortcut
- For polymorphic data, use `oneof` (Protobuf), discriminated union (OpenAPI `oneOf` with `discriminator`), or union type (GraphQL, Avro)
- For SQL columns storing structured data, define a JSON Schema or application-level validation enforced on every write
- If `additionalProperties` is needed, constrain it: `additionalProperties: { type: string }` is better than `true`
- Treat every use of `Any`/`Struct`/`bytes`-for-structured-data as technical debt

### SCHEMA-ENUM-EVOLUTION — Enums must be safe to extend; consumers must handle unknown values

Every enum in a published schema must include an explicit unknown or unspecified sentinel value, and every consumer must handle enum values it does not recognise without crashing or discarding the containing message. Adding a new variant is routine schema evolution; if any consumer treats an unknown variant as an error, enum extension becomes a breaking change.

Why it matters:

Enums are one of the most common sources of subtle schema incompatibility. When a new variant is added, producers emit it immediately after deployment. If a consumer running the old schema has no fallback — because the enum is closed or deserialization throws on unknown values — the consumer fails on valid data.

Good practice:

- In Protobuf, always define the first enum value (field number 0) as `FOO_UNSPECIFIED = 0;`
- In OpenAPI, document every enum as non-exhaustive and include an `UNKNOWN` or `OTHER` variant
- In Avro, set the `default` property on every enum to the sentinel symbol
- In application code, always include a `default` / `_` / `else` branch in every `switch` or `match` on an enum
- Treat a missing `default` case on an enum switch as a bug — it is a forward-compatibility hazard
