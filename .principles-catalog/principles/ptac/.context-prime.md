# Plain-Text-as-Code principles by flemming-n-larsen.

### PTAC-PLAIN-TEXT-FIRST

**Principle:** Choose plain text representations for all artifacts that humans or tools need to read, review, or modify. Binary formats should be the exception, not the default — use them only when plain text is technically infeasible.

**Why it matters:** Binary formats create friction: they can't be diffed, reviewed in a pull request, searched with grep, or read without a proprietary tool. Plain text artifacts can be version-controlled, merged, audited, and processed by AI tools without conversion.

**Good practice:** Prefer `openapi.yaml` over `openapi.json.gz`; `architecture.mermaid` over `architecture.vsdx`; `README.md` over `README.docx`.

### PTAC-COMPOSABLE-FILES

**Principle:** Each file should express one coherent concept or concern. Compose larger systems by combining small, focused files rather than packing everything into a single large file.

**Why it matters:** Large monolithic files are hard to navigate, review, and reason about — for humans and AI tools alike. Small, composable files can be read in full, reviewed in isolation, and combined without understanding the whole.

**Good practice:** Split a large `infra/main.tf` into `network.tf`, `compute.tf`, `storage.tf`, and `iam.tf`. Split a monolithic `everything.md` into `quickstart.md`, `reference.md`, and `architecture.md`. Files over 300 lines are a signal to reconsider structure.

### PTAC-NO-GENERATED-BLOBS

**Principle:** Version control tracks *intent*, not *output*. Commit the source files and the instructions to generate them; let the build system produce the output. Generated files that are committed must be clearly marked, kept in sync, and justified by a concrete consumer need.

**Why it matters:** Generated files committed without discipline cause constant noise: every regeneration produces a diff with no semantic content. When source and generated output diverge, the repo tells a lie.

**Good practice:** Commit `src/api/openapi.yaml` and `scripts/generate-client.sh`, not the generated `ApiClient.java`. If generated output must be committed (e.g., for consumers without build tools), mark it with `# Generated by ... — do not edit manually`.

---

### PTAC-LEAN-CONTENT

**Principle:** Keep plain-text content DRY and front-loaded; every line must earn its place. Put the most critical information first — AI recall degrades for content buried mid-document. Link to one canonical source rather than repeating the same fact across files.

**Why it matters:** Token budgets are finite and costly. Verbose documents inflate AI context, increase error rates, and slow human reviewers. DRY documents reduce maintenance: the same fact in two places creates two divergence risks.

**Good practice:** Front-load the decision, constraint, or summary in the first three lines. Cut filler phrases ("This section describes...", "As mentioned above..."). Files over 200 lines are a signal to split or trim. Test: can the first three lines tell an AI what this document is about?

---

### PTAC-NAVIGABLE-DIRS

**Principle:** Every plain-text directory must contain two files: `INDEX.md` — a structured list of subdirectories and files with one-line descriptions (for AI tools and automation) — and `README.md` — a human-oriented description of the directory's purpose (auto-rendered on GitHub and other git-hosting platforms).

**Why it matters:** Without navigation aids, AI tools must speculatively read every file to understand the structure, wasting context. Human reviewers can't quickly orient themselves. Both files serve distinct roles: README explains *what this is*; INDEX says *what's in here and where to find it*.

**Good practice:** `INDEX.md` uses a table (`| Path | Description |`) listing every subdirectory and file. `README.md` describes purpose, scope, and how to use the content. Keep them separate — don't merge them into a single file.
